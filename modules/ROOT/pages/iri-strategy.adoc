== IRI patterns

<<x,Resources>> and their representations are identified by IRIs, each of which must be of the following form:

.Base IRI pattern
[source,subs="attributes"]
....
https://\{kind}.{domain}/\{category}/\{namespace}[/\{version}][/\{reference}]
....

[horizontal,labelwidth=25%]
`\{kind}`::
    `data` | `id` | `www` +
`\{category}`::
    `data-product` | `doc` |`ontology` | `register` | `schema` | `thesaurus`

`\{namespace}`::
    Path which encodes the namespace of the component. This can be nested as deeply as necessary, and has no formal (nor machine-readable) meaning.

`\{version}`::
    Version specifier (if applicable).

`\{reference}`::
    Local name of some referrent in the namespace (if applicable).

.Resource kinds
****
Traditionally, all resources were web documents of some kind or another, and the URLs locating them can serve as their identifying IRIs. More formally, those kinds of resources are called _information resources_. In this document we further distinguish two types of information resources:

* documentation information resources (indicated by the `www` subdomain)
* data information resources (indicated by the `data` subdomain)

Since the dawn of the Semantic Web, however, we want anyone to be able to say anything about anything. This includes the ability to make statements about entities or concepts in the real world. Such resources, however, are not information resources and are not on the web. To be able to state things about them, they need to have IRIs, but since they are not information resources, they are recognized as their own resource kind with a dedicated subdomain:

* real-world entities or concepts (indicated by the `id` subdomain)

Note there can be information representations which represent another resource. For example,

=== Registers

A register is a logical container of resources which represent entities from the real-world domain. Each register contains one type of entity.

.IRI pattern for a register
[source,subs="attributes"]
....
https://data.{domain}/register/\{namespace}
....

.IRI pattern for a register entity
[source,subs="attributes"]
....
https://id.{domain}/register/\{namespace}/\{reference}
....

[cols="h,2,2",width=70%]
|===
| | Register | Entity

| Resource kind
a| (`data`) data information resource
| (`id`) real-world entity

| Category
2+| `register`

| Namespace
2+| Namespace identifying the register

| Version
2+|_n/a_

| Reference
| _n/a_
| Reference name (local to the register) of the contained entity
|===

.IRI pattern for a model
[source,subs="attributes"]

https://data.{domain}/ontology/\{namespace}[/{version}]

.IRI pattern for a model element
[source,subs="attributes"]
....
https://id.{domain}/ontology/\{namespace}[/{version}]/\{reference}
....

[cols="h,2,2",width=70%]
|===
| | Model | Element

| Resource kind
a| (`data`) data information resource
| (`id`) real-world concept

| Category
2+| `ontology` \| `thesaurus` \| `schema`

| Namespace
2+| Namespace identifying the model

| Version +
_(optional)_
a| Model version, following the pattern:

`X.Y.Z` +

(where `X, Y, Z` are non-negative integers)
a|_n/a_

| Reference
| _n/a_
| Name (local to the model) of the element
|===

=== Documentation

Documentation intended for reading by humans, not machines. A documentation project can consist of a mere single-page document, but also be comprised of a complex nested structure containing many pages and potentially many layers of organisation.

NOTE: Do not confuse the category _documentation_ with documentation representations as obtained through using `www` IRIs. See also: {fn-doc-iri}

.IRI pattern for a project
[source,subs="attributes"]
....
https://www.{domain}/doc/\{namespace}[/{version}]
....

.IRI pattern for a part
[source,subs="attributes"]
....
https://www.{domain}/doc/\{namespace}[/{version}]/\{reference}
....

[cols="h,2,2",width=70%]
|===
| | Project | Part

| Resource kind
2+| (`www`) documentation information resource

| Category
2+| `doc` footnote:doc-iri[]

| Namespace
2+| Namespace identifying the project

| Version +
_(optional)_
a| Project version, following the pattern:

`X.Y.Z` +

(where `X, Y, Z` are non-negative integers)
a|_n/a_

| Reference
| _n/a_
| Name (local to the project) of the part (e.g. page)
|===

=== Data product

Special category which contains data products.

NOTE: This is much like a register, but data product are information resources whereas other registers are not. Therefore, to avoid confusion, and because data products are important, this special category has been introduced.

.IRI pattern for a data product
[source,subs="attributes"]
....
https://data.{domain}/data-product/\{reference}[/{version}]
....

[cols="h,2",width=70%]
|===
| Resource kind
| (`data`) data information resource

| Category
| `data-product`

| Namespace
| _n/a_

| Version +
_(optional)_
a| Data product version, following the pattern:

`X.Y.Z` +

(where `X, Y, Z` are non-negative integers)

| Reference
| Name of the data product
|===

== Resource representations

Any resource can have several representations associated with them. Each of those representations are themselves information resources, either of the data or documentation kind.

=== Real-world entities and concepts

Resources which represent real-world entities or concepts are not information resources and therefore do not have an inherent representation on the web. However, other information resources can represent them.

.Content negotiation for real-world entitiy
image::conneg-real-world-entities.png[]

A `GET` request to the `id` IRI can utilize _content negotiation_ to clarify what type of representation is desired. The web server can then use a _303 See Other_ redirect to the URI of an information resource which provides appropriate representation.

.IRI Strategy
====
Let's consider the IRI for the _IRI Strategy_ document.

.Generic IRI
[source,subs="attributes"]
....
https://www.{domain}/doc/modeling-guide/iri-strategy
....

Note that it is part of the _Modeling Guide_ documentation project, which would act as its container.

To request a specific version of the _Modeling Guide_ one could use the following IRI:

.Versioned IRI
[source,subs="attributes"]
....
https://www.{domain}/doc/modeling-guide/1.1.0/iri-strategy
....

Note that it is not supported to version the _IRI Strategy_ document itself, only the project it is contained in. This is analogous to the inability to version ontology terms.

.IRI to Specific page
[source,subs="attributes"]
....
https://www.{domain}/doc/modeling-guide/iri-strategy/decisions
....

If documentation consists of multiple pages, these can be referenced simply by expanding the path as illustrated in this example.

[%collapsible]
====
[source,yaml,subs="+attributes"]
....
id: https://data.{domain}/schema/data-product/transitietechnieken <1>
version: 1.0.1 <2>
name: transitietechnieken <3>
prefixes: <4>
  nbnl: https://id.{domain}/taxonomy/energiesysteembeheer/
  cimnl: https://id.{domain}/ont/cim-nl/
classes:
  RegisteredResource:
    class_uri: cim:RegisteredResource
    attributes:
      resource_type:
        slot_uri: cimnl:RegisteredResource.resourceType <5>
        range: ResourceKind
        exact_mappings:
          - nbnl:soortInstallatie <6>
....
<1> The generic IRI of the schema, i.e. not a specific version.
<2> The version of the schema this document represents.
<3> Technical name of the schema. This has local scope, but it is good practice to keep it equal to the model name in the IRI.
<4> Prefixes for easy referring to terms from other models.
<5> Referring to a term from another ontology.
<6> Referring to a term from another thesaurus.
